# リポジトリ固有の指示

このファイルは、すべてのコーディング AI および人間が遵守しなければならないリポジトリの運用についての説明です。

## コミット

コミットのタイトルや説明文は、原則として日本語で書いてください。

PR 内におけるコミット履歴は、トレーサビリティのため、原則として、force push の代わりに、打ち消しのコミットをしなければなりません。
ただし、著作権などの問題のあるコンテンツは、コミット履歴から排除しなければなりません。

## ソースコード

コメントは、原則として日本語で書いてください。
例外メッセージのような、プログラムの実行中に外部から観測可能なテキストは、すべて英語で書いてください。

リポジトリ内の既存コードをよく見て、あらゆる部分の記述スタイルを一貫させてください。

変数名には、基本的な型や、一部の例外的な事例を除き、`ctx: Context` や `str: String` といった、クラス名から乖離した名前を付けてはいけません。
for 文は極力使わず、forEach 系の拡張関数を使ってください。
`Pair(a, b)`（2要素タプル） と `a to b`（キーと値） は、意味論的に使い分けてください。

## テストコード

Xarpite にはテストコードが存在します。
すべての機能はテストコードを伴い、テストを通過しなければなりません。
テストコードは、ドキュメント類において規定されるすべての仕様を丁度過不足なく検証しなければなりません。

テストコードは、すべての部分が LLM によって管理されており、人間はケアをしていません。
テストコードの編集は、それが妥当な改善である限り、自発的かつ積極的かつ大胆かつ躊躇なく行って構いません。

テストコードの編集、コミット、PR 上への push は、明示的に禁止されていない限り、原則として、人間による承諾なしで実行してかまいません。

### 深すぎる再帰によるネイティブクラッシュ

Xarpite の言語レベルの関数呼び出しは、評価のたびにホスト言語のコールスタックを実際に1段ずつ消費します。
Kotlin/Native は末尾呼び出しの最適化を行わないため、数千段に及ぶ深い再帰をテスト入力として評価すると、ネイティブのコールスタックが枯渇し、テスト実行バイナリが SIGSEGV でハードクラッシュします。
これは実行環境やタイミングに依存して非決定的に発生するため、CI における断続的な失敗の原因となります。

そこで、テストには、ネイティブのコールスタックを枯渇させうるほど深い再帰呼び出しや、それに準ずる深いネストを、テスト入力として用いてはいけません。

### Windows 上での jvmTest における EXEC 呼び出しの問題

Windows 環境で jvmTest を行うと、純粋な Windows の環境内で Xarpite のコードが走ります。
これにより、Linux と Windows でのコマンド引数の内部的な扱いの違いに起因する、Windows 版 Java の外部プロセス呼び出しライブラリの環境依存的なバグに苛まれます。
そこで、`EXEC` 等の外部コマンドの呼び出しを伴うテストは、必ず、下記の手続きで記述してください。

---

コマンドの実行を、「コマンドそのものの呼び出し」ではなく、「コマンド呼び出しを含むシェルスクリプトの実行」とみなしてください。
それを `EXEC("bash", "-c", ...)` のように、一旦 `bash -c` を仲介させて実行してください。
これにより、環境側が提供するコマンドのレパートリーや、コマンドライン引数の内部処理方式の違いが吸収されます。

シェルスクリプト `...` の部分は完全に自由に記述することはできず、`$` を含むいくつかの記号が、バグった展開処理によって壊れます。
そこで、シェルスクリプトの UTF-8 バイト列を 16 進エンコードすることにより、記号類を含まない形にエンコードしてください。
そして、エンコードしたそれをその場でデコードし、bash コマンドに渡して実行するようなシェルスクリプトを記述してください。

まとめると、おおよそ以下のようになります。

```kotlin
val script = "date -u --iso=s"
val hex = script.encodeToByteArray().toHexString()
val xarpiteCode = """EXEC("bash", "-c", %>xxd -r -p <<<'$hex' | bash<%)"""
```

## ビルドと実行

`jvmTest` 等の個別のテスト用 Gradle タスクは、あなたに誤った情報を与えるだけで、何の役にも立ちません。
たとえどのような些細な編集であろうとも、テストには必ず `build` タスクを常に使用してください。
さもなくば、あなたは真の失敗原因を見落とし、事実誤認に基づいてソースコードを徹底的に「修正」して回り、PR を破壊します。

## ドキュメント

このプロジェクトには、`/pages/docs` 以下に言語のドキュメントが存在します。

ドキュメントには複数の言語版が存在し、それらは意味論的に過不足なく一致する内容でなければなりません。
ドキュメントは日本語が「正」であり、他言語版はその「翻訳版」です。
ドキュメントは流暢かつ自然な言葉で記述されていなければなりません。

### 記述範囲

以下の内容は、過不足なく網羅的に記載されなければなりません。

- 言語の利用者が意識すべき、言語機能のすべて

---

以下の内容をドキュメントに含めてはいけません。

- 言語の利用者が意識する必要のない、純粋に内部的な識別子名やメカニズム
- ビルドツールの使用方法など、言語機能に該当しないもの

### 内容

ドキュメント内の構造は、そのファイルが扱う対象の実態に基づいて、合理的である必要があります。
その結果、複数のドキュメントファイルの間で構造が非一貫的となることは、許容されます。

ドキュメント内の項目は、追加順や辞書順などではなく、論理的および意味論的に合理的で一貫した順序である必要があります。

細かいレベルでのあらゆる記述スタイルは、可能な限りドキュメント全体を通して一貫していなければなりません。

ドキュメントを記述する際は、既存部分を熟読し、粒度、構造、体裁、そのたあらゆる記述スタイルを極力一貫させなければなりません。
その記述は何に紐づいた説明で、どのセクションのどの位置に記述されるべきかを、入念に検討してください。

## 更新履歴

`CHANGELOG.md` の更新は断片ファイル群 `/changelog.d/${任意のタイトル}.md` に基づいて機械的に行われます。
プルリクエストは `CHANGELOG.md` を直接編集しないでください。

断片ファイルは 0 個以上の末尾改行のある行で構成されます。
互いに独立した変更は別々の行に記述しなければなりません。
プルリクエストは（中身が空であっても）丁度1個の断片ファイルを持たなければなりません。

行は大文字始まりの過去形の英文による簡潔な更新点の説明を持ちます。
内部的な改善や実装手段などのユーザーから見えないような記述を含めてはいけません。
識別子名やコード断片は Markdown のインラインコード記法を使用してください。
記述例は既存の `CHANGELOG.md` を参照してください。

行には変更 `!Changed ...`、改善 `Added ...`、修正 `?Fixed ...` の区別があり、それぞれ異なる prefix で始まります。

```
いい例: ?Fixed an error when passing an absolute path to the `READ` function.
悪い例: I created AbsolutePathReader and refactored the read function to use it, because ArrayIndexOutOfBoundsException occurs due to a bug in parsing absolute paths in the read function.
```

## ビルド用 Xarpite ランタイム

`scripts/xarpite` 内には、ビルドツールの一部としての Xarpite ランタイムがコミットされています。
これはこのリポジトリの成果物の一部ではなく、同封されている正規のアップデートスクリプト以外の手段で編集してはいけません。
