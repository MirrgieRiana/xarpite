# Issue #177: インデントによるラムダ渡し — 考察メモ

## 1. 提案されている構文の整理

Issue #177では3つの構文案が提示されている：

```
LAUNCH ( => |

```

### 案B: 行末コロン `:` (Python方式)
```
LAUNCH ( => :
  OUT << "Hello, World!"
```

### 案C: ヒアドキュメント方式 `<<`
```
LAUNCH ( => << )
```

```
a:
    b

```
→ `a` に暗黙の括弧で `b; c` を渡す

### 案E: 演算子の後にコロンを付ける
```
a ||:
    OUT << b
    return!! c
```

### 案F: `->`の後にコロンを付ける (変数宣言)
```
f := args ->:
    a
    b
```

## 2. 現在の `:` の使用箇所

Grammar.ktにおける `:` の使用を完全に列挙する：

| 構文 | パーサー定義 | 優先順位レベル | 意味 |
|------|-------------|---------------|------|
| `::` | `rightOperator` (L169) | 後置演算子 | メソッドバインド |
| `?::` | `rightOperator` (L170) | 後置演算子 | Null安全メソッドバインド |
| `!:` | `executionOperator` (L252) | 実行演算子 | ラベル定義 |
| `:=` | `assignmentOperator` (L257) | 代入演算子 | 変数宣言 |
| `:` | `assignmentOperator` (L256) | 代入演算子 | エントリー演算子 |
| `?:` | `condition` (L233) | 条件演算子 | Elvis演算子 |
| `? ... :` | `condition` (L232) | 条件演算子 | 三項演算子 |

**パースの判別方法：**
- `assignmentOperator` での `:` は `-':' * !'=' * !':' ` つまり「`=` でも `:` でもないものが後続する `:` 」
- `condition` での `:` は `? body :` のパターンの一部
- 三項演算子内の `:` は `-':' * !':'` つまり「`::` でない `:` 」

## 3. 現在の括弧構文の構造

### 括弧リテラル（独立した括弧）
```
arrowRound:  (args => body)     → BracketsLiteralArrowedRoundNode
arrowSquare: [args => body]     → BracketsLiteralArrowedSquareNode
arrowCurly:  {args => body}     → BracketsLiteralArrowedCurlyNode
round:       (body)             → BracketsLiteralSimpleRoundNode
square:      [body]             → BracketsLiteralSimpleSquareNode
curly:       {body}             → BracketsLiteralSimpleCurlyNode
```

### 後置括弧（前に式がある括弧）
```
f(args => body)    → BracketsRightArrowedRoundNode
f[args => body]    → BracketsRightArrowedSquareNode
f{args => body}    → BracketsRightArrowedCurlyNode
f(body)            → BracketsRightSimpleRoundNode
f[body]            → BracketsRightSimpleSquareNode
f{body}            → BracketsRightSimpleCurlyNode
```

**重要な観察:**
- 3種の括弧 `()`, `[]`, `{}` × 2スタイル（arrow/simple）× 2位置（literal/right）= **12種のノード型**
- インデント括弧を追加すると、最低でもさらに6〜12種のノード型が必要になる可能性がある

## 4. 空白・改行の処理方式

現在のGrammarは**インデント非感知**である：

```kotlin
val br: Parser<Char> = +Regex("""\n|\r\n?""") map { '\n' }
val s: Parser<Tuple0> = -(+Regex("""[ \t]+""") + lineComment + blockComment).zeroOrMore
val b: Parser<Tuple0> = s * -(br * s).zeroOrMore
```

- `s` = 水平空白（スペース・タブ・コメント）のみ
- `b` = 水平空白 + 任意個数の改行+水平空白
- `br` = 改行トークン

`b` は改行を含む空白を自由に消費するため、現状では「行末」という概念がパーサーに存在しない。

## 5. 「行末コロン」が既存コードを壊すケースの分析

### 5.1 三項演算子の連鎖

```xarpite
method == "apple"  ? (() -> "Fallback apple") :
method == "banana" ? (x, y, z -> "Fallback") :
method == "cherry" ? (() -> !!"ERROR") :
                     NULL
```

ここでは `:` が行末に来ている。`:` の直後に改行があり、次の行にインデントされた式が来る。
**これは既存の三項演算子の合法的な使い方であり、インデント括弧と区別不可能。**

### 5.2 オブジェクトリテラル内のエントリー

```xarpite
{
    a: 1
    b: 2
    c: 3
}
```

ここでも `:` の後に値が同一行にあるが、値が複雑な場合は次の行に折り返される可能性がある：

```xarpite
{
    longKeyName:
        someVeryLongExpression + anotherExpression
}
```

この形式は現在のパーサーでは合法（`b` が改行を許容するため）。しかし、コロンインデント構文を導入すると、エントリー値がインデント括弧として解釈されてしまう。

### 5.3 変数宣言 `:=`

```xarpite
f := args ->:
    a
    b
```

Issue #177のコメントでこの形式が提案されているが、現在 `:=` は単一トークンとしてパースされる。`->:` も同様に `->` と `:` に分解可能だが、現在のパーサーでは `->` が代入演算子レベルで消費され、その右辺は `stream` を期待する。

### 5.4 Elvis演算子 `?:`

```xarpite
value ?:
    fallback_expression
```

Elvis演算子の後に改行+インデントで代替値を書くケースは現在合法。

## 6. 案Fの詳細分析：`:` をfactorとして導入する案

Issueの最新コメントで言及されている：

> `:` を頭に持つインデント括弧が定義できればいい。ただ、`:` は頭に来ないという前提が崩壊する

これは `:` を**式の先頭（factor位置）に配置する**のではなく、**演算子の直後に配置する**構文を意味する。

### 6.1 構文パターンの再整理

提案を分解すると、`:` + インデントブロックで「暗黙の括弧」を表現する：

```
:           →  (              （丸括弧、コード実行スコープ）
:NEWLINE    →  開始
INDENT      →  内容
DEDENT      →  )              （閉じ括弧）
```

ただし、括弧の種類（丸、角、波）を区別できない。
Issue本文の提案では開き括弧を明示する（`(`、`[`、`{`）ので、コロンは「閉じ括弧を省略してインデントで代用する」というマーカーに過ぎない。

### 6.2 `:` が factor 位置に来る場合の影響

現在 `:` は factor 位置に来ない。`factor` パーサーは以下で構成される：

```
jump + hexadecimal + identifier + quotedIdentifier + float + integer +
rawString + templateString + embeddedString + regex + brackets
```

ここに `:INDENT_BLOCK` を追加すると、以下のパースに影響する：

1. **`assignmentOperator` の `:` との競合**: `a : b` の `:` が演算子なのかインデント括弧なのか
2. **三項演算子 `? ... :` との競合**: 三項演算子の `:` の後にインデントが来た場合
3. **`!:` との競合**: ラベル定義の `!:` の `:` 部分

### 6.3 提案の核心的問題

> `:` は頭に来ないという前提が崩壊する

この「前提」の意味を明確にする：

現在、`:` が**式の一部として**登場するのは常に**中置演算子**としてである。つまり、左辺が必ず存在する。
`:` を factor（式の先頭）として導入すると、パーサーは `:` を見た瞬間に「これは中置演算子の `:` なのか、factor としてのインデント括弧なのか」を判断しなければならない。

**PEGパーサーでは先に定義された選択肢が優先される**ため、factorに `:INDENT_BLOCK` を追加しても、中置演算子 `:` が先に消費されるなら問題ないように見える。しかし：

- `a :` の後に改行+インデントが来た場合、中置演算子 `:` の右辺として解釈される（現状の動作）
- `:` が行頭・式頭に来た場合のみインデント括弧として解釈する

→ これならば**前方互換性がある**可能性が高い。

## 7. パーサーにインデント感知を導入する方法論

### 7.1 方式A: レキサーレベルのインデント変換（Python方式）

入力テキストをパースする前に、インデントレベルの変化を `INDENT`/`DEDENT` 仮想トークンに変換する前処理を行う。

**利点:**
- パーサー本体の変更が最小限
- 実績のある手法（Python, Haskell, etc.）

**欠点:**
- Xarpiteは改行をセミコロン代わりに使うが、改行の意味が文脈依存になる
- 括弧内の改行は現在自由に入れられる（`b` が改行を消費する）ため、括弧内のインデント変化をどう扱うかが問題
- 前処理段階でコロンの意味を判断するのが困難

### 7.2 方式B: パーサー内でインデントレベルを追跡する

パーサーの状態として現在のインデントレベルを保持し、`:` + 改行を検出したらインデントブロックのパースモードに入る。

**利点:**
- 文脈に応じた正確なインデント処理が可能
- 既存の `b` パーサーの動作を変更せずに済む可能性がある

**欠点:**
- xarpeg PEGパーサーライブラリが状態の保持に対応しているか不明
- バックトラッキングとの相性が悪い可能性がある

### 7.3 方式C: コロンの直後に特殊トークンを挿入する簡易方式

`:` + 改行 + インデント増加を検出し、そのブロック全体を仮想的な括弧で囲む前処理を行う。

```
入力:
f := args ->:
    a
    b

変換後:
f := args -> (
    a
    b
)
```

**利点:**
- パーサーの変更が不要
- 前処理段階で完結する
- 既存の括弧構文をそのまま再利用できる

**欠点:**
- 括弧の種類（`()`, `[]`, `{}`）を前処理で判別する必要がある
- 直前の文脈を見てどの括弧に変換するか決める必要がある
- エラーメッセージの行番号がずれる可能性がある

## 8. 案ごとの実現可能性評価

### 案B+F統合: `演算子:` + インデントブロック

```xarpite
# 変数宣言
f := args ->:
    a
    b

# パイプ
a ||:
    OUT << b
    return!! c

# 通常のコロン（エントリー演算子）+ インデント
a:
    b
    c
```

**パース戦略:**

`:` が **改行の直前**（水平空白・コメントを除く）に来た場合、次の行からのインデントブロックを開始する。

判別条件：
1. `:` の直後に（水平空白を除いて）改行がある → インデント括弧
2. `:` の直後に（水平空白を除いて）非改行の式がある → 通常のエントリー演算子

**問題点:**

三項演算子の連鎖で `:` が行末に来るケース：
```xarpite
condition1 ? value1 :
condition2 ? value2 :
             value3
```

これは現在 `condition` パーサーで処理される。三項演算子内の `:` はインデント括弧ではないが、上記の判別条件に一致してしまう。

**解決策の候補:**
1. 三項演算子内の `:` は先にパースされるため、コロンインデントは代入演算子レベル以下でのみ有効にする
2. パーサーの優先順位により、三項演算子内の `:` は `condition` パーサーが消費し、残らない
3. 「`:` の直前が `?` である場合はインデント括弧にしない」というルールを追加する

**実は、PEGパーサーの特性上、問題にならない可能性がある：**

三項演算子 `cond ? val : alt` の `:` は `condition` パーサーの一部として消費される。`condition` パーサーは `stream` より優先順位が高い。つまり、`stream` → `commas` → `condition` の順で評価される中で、三項演算子の `:` は `condition` 内で処理される。

一方、エントリー演算子の `:` は `assignmentOperator`（`stream` レベル）で処理される。

→ **パーサーの優先順位により、三項演算子の `:` とインデント括弧の `:` は自然に区別される**。

ただし「`:` の後にインデントブロックを開始する」という処理が `assignmentOperator` レベルにある場合、三項演算子の `:` はそれより先に消費されるので衝突しない。

## 9. 「`:` を頭に持つインデント括弧」の具体的な文法定義案

### 9.1 `:` がfactor位置に来る案

```
colonIndentBlock ::= ':' NEWLINE INDENT expression DEDENT
factor           ::= ... | colonIndentBlock
```

この場合、`:`は他のfactorと同列に扱われる。
`a :` の `:` は中置演算子として消費され、factor位置の `:` とは競合しない。

**使用例:**
```xarpite
f := args ->:
    a
    b
# パース: f := args -> :(BLOCK a; b)
# :BLOCK は (a; b) と等価
```

ここで `->` は代入演算子で、右辺に `stream` を期待する。`stream` → `commas` → ... → `factor` の解決で、`:BLOCK` がfactorとしてマッチする。

### 9.2 `:` が後置演算子の一部として来る案

```
rightOperator ::= ... | ':' NEWLINE INDENT expression DEDENT
```

この場合、`a:` のように直前の式に対する後置演算子として `:BLOCK` が作用する。

**使用例:**
```xarpite
LAUNCH ( =>:
    OUT << "Hello, World!"
)
# パース: LAUNCH (=>:BLOCK(OUT << "Hello, World!"))
```

しかしこれは後置括弧と何が違うのか？ 実質的に `LAUNCH ( => (OUT << "Hello, World!") )` と等価であり、閉じ括弧が省略できるだけのメリットしかない。

### 9.3 演算子の後にfactorとして来る案（推奨）

`factor` にインデントブロックを追加する方式が最もシンプル：

```kotlin
val indentBlock: Parser<Node> = -':' * -br * INDENT * ref { expression } * DEDENT map { ... }
val factor: Parser<Node> = ... + indentBlock
```

しかし、`:`が中置演算子としても使われるため、factorとして`:` を消費しようとすると先に中置演算子として消費される問題がある。

**回避策:** インデントブロックの `:` は必ず「改行直後のインデント増加」を伴うため、中置演算子の `:` とは区別可能。中置演算子の `:` は右辺の式を同一行で開始する。

## 10. 実装の技術的ハードル

### 10.1 xarpeg PEGパーサーでのインデント感知

xarpegがPEG（Parsing Expression Grammar）ベースのパーサーコンビネータであることから：

- **バックトラッキング**: PEGはバックトラックするため、インデント状態の管理が複雑になる
- **状態の保持**: PEGパーサーは一般的にステートレスだが、インデントレベルの追跡には状態が必要
- **xarpegの機能**: `mapEx` でパース結果にコンテキスト情報を付与できることから、何らかの状態管理は可能かもしれない

### 10.2 前処理方式のトレードオフ

前処理でインデントを括弧に変換する方式（方式C）が最も実装が簡単だが、以下のトレードオフがある：

**括弧の種類の決定:**
- `->:` → `-> (...)` (丸括弧、コードブロック)
- `||:` → `|| (...)` (丸括弧、コードブロック)
- `LAUNCH (:` → `LAUNCH ((...)` (丸括弧)
- `:` 単独 → `(...)` (丸括弧がデフォルト？)

→ **常に丸括弧 `()` に変換する**のが最もシンプル。Xarpiteでは `()` がコードブロック相当であり、変数スコープを作る。

**問題:** オブジェクトリテラルをインデントブロックで書きたい場合は `{}` が必要だが、`:` だけでは括弧の種類を指定できない。

### 10.3 括弧の種類を指定する構文の検討

```
:( ... )   → 丸括弧（コードブロック/スコープ）
:[ ... ]   → 角括弧（配列）
:{ ... }   → 波括弧（オブジェクト）
```

しかしインデントブロックの利点は閉じ括弧が不要なことなので：

```
:(         → インデント丸括弧
:[         → インデント角括弧
:{         → インデント波括弧
```

**あるいは開き括弧の種類で判別する：**

Issue本文の案A/B:
```
LAUNCH ( => :
  OUT << "Hello, World!"
```

ここでは `(` が明示されているので、`:` は「この括弧をインデントで閉じる」というマーカーとして機能する。

## 11. 統合的な設計案

### 設計案1: 「コロン＝次行インデントブロック開始」

**ルール:**
1. `:` の直後に（水平空白を除いて）改行がある場合、インデントブロック開始
2. インデントブロックは丸括弧 `()` として扱われる
3. 三項演算子の `:` は `condition` レベルで消費されるため影響なし
4. エントリー演算子の `:` は同一行に右辺がある場合に限る

**影響範囲:**
- エントリー演算子 `:` の右辺が改行後にある既存コードが壊れる
  - 例: `key:\n    value` → 現在は `key: value` と等価、変更後は `key:(value)` と解釈
- ただし、このような書き方は実際にはほとんど使われない可能性が高い

### 設計案2: 「新演算子を導入する」

`:` の代わりに別の記号を使う：

```xarpite
f := args ->|
    a
    b
```

**利点:** 既存のコロン使用と完全に非互換がない
**欠点:** 直感的でない、記号の選択が限られる

### 設計案3: 「`#:`による明示的なインデント括弧」

```xarpite
f := args -> #:
    a
    b
```

`#:` は新しい2文字トークンとして導入する。`#` は現在コメントの開始だが、`#:` はコメントではない新しいトークンにできる。

**問題:** `#` がコメント開始としてレキサーに認識されるため、`#:` の前に `#` がコメント開始として消費される可能性がある。

→ コメントのパーサーを修正して `#:` を除外する必要がある。これは既存のコードに影響する。

## 12. `:` を頭に持つインデント括弧がfactorとして機能する場合の既存コードへの影響

### 12.1 パーサーの優先順位による自然な分離

Xarpiteのパーサー階層：
```
factor → right → pow → left → mul → add → range → infixIdentifier → match →
spaceship → comparison → and → or → condition → commas → stream(pipe/execution/assignment) →
semicolons → expression
```

- 三項演算子 `? :` は `condition` レベル
- エントリー演算子 `:` は `assignmentOperator` → `stream` レベル
- factorは最も高い優先順位

**つまり、`:` がfactor位置（式の先頭）に来る場合、中置演算子の `:` とは構造的に異なる位置にある。**

中置演算子の `:` は常に左辺が存在する。factor位置の `:` には左辺がない。
PEGパーサーでは、`a : b` と書いた場合、`a` がfactorとしてパースされ、`:` が中置演算子としてパースされ、`b` がその右辺としてパースされる。
factor位置の `:` は、式の先頭に `:` が来たときのみマッチする。

### 12.2 実際に factor位置に `:` が来るケース

factorが期待される位置は：
1. 式の先頭
2. 中置演算子の右辺
3. 前置演算子の後
4. 括弧内の式

**例1: 式の先頭**
```xarpite
:
    a
    b
```
→ `:INDENT_BLOCK(a; b)` ≡ `(a; b)`

**例2: 代入の右辺**
```xarpite
f := :
    a
    b
```
→ `f := :INDENT_BLOCK(a; b)` ≡ `f := (a; b)`

**例3: `->` の右辺**
```xarpite
f := args -> :
    a
    b
```
→ `f := args -> :INDENT_BLOCK(a; b)` ≡ `f := args -> (a; b)`

**これは `->`  の後の空白(s) + `:` + 改行(br) + インデント**

ただし！ここで問題がある。`:` は `assignmentOperator` で中置演算子として定義されている。
`args -> : INDENT` の場合、`->` は代入演算子として `args` と右辺を結合する。右辺には `stream` が来る。`stream` → `commas` → ... → `factor` の解決で、`:` がfactorとしてマッチする。

**つまり `->:` のように空白なしで書くと:**
- `->` が代入演算子として消費される
- `:` が factor 位置に来る
- factor `:` + 改行 + インデントでインデントブロックが開始

**空白ありで `-> :` と書いても同じ。** `->` が消費された後、`s`（水平空白）が消費され、factorとして `:` が試行される。

### 12.3 潜在的な競合

**問題ケース: `a : b` でbが改行後にある場合**

```xarpite
key :
    value
```

現在のパーサーの動作：
1. `key` がfactorとしてパースされる
2. `:` が `assignmentOperator` の中置演算子としてパースされる
3. 右辺の `value` が（`b` により改行を含む空白を消費した後に）パースされる
4. 結果: `InfixColonNode(key, value)` ≡ `[key, value]`

`:` をfactor位置に追加した場合の動作：
1. `key` がfactorとしてパースされる
2. `:` が `assignmentOperator` の中置演算子としてパースされる
3. **中置演算子 `:` が先にマッチするため、factor位置の `:` は試行されない**
4. 結果: 現状と同じ `InfixColonNode(key, value)`

→ **この場合、競合は起きない。** 中置演算子が先に消費される。

**問題ケース: `:` が式の先頭に来る場合**

```xarpite
result := :
    a
    b
```

1. `result` がfactorとしてパースされる
2. `:=` が `assignmentOperator` としてパースされる
3. 右辺の先頭で `:` をfactorとして試行 → マッチ（改行+インデントが後続）
4. 結果: `InfixColonEqualNode(result, IndentBlockNode(a; b))`

→ **正しく動作する。**

---

## 13. 結論と推奨

### 推奨設計: `:` をfactorとして追加する方式

**構文:** `:` + 改行 + インデント増加 → `(expression)` として解釈

**理由:**
1. 中置演算子 `:` との競合がPEGの特性により自然に解消される
2. 最小限のパーサー変更で実現可能（factorに1項追加するだけ）
3. 括弧の種類は常に丸括弧 `()` で統一（最もシンプル）
4. 既存コードとの後方互換性が高い

**残課題:**
1. xarpeg PEGパーサーでインデントレベルの追跡をどう実現するか
2. 括弧の種類（`()`, `[]`, `{}`）を指定したい場合の拡張構文
3. エラーメッセージの品質
4. エディタのシンタックスハイライト対応

### 括弧種類の拡張案

将来的に括弧の種類を指定したい場合：
```
:(  → 丸括弧インデントブロック（デフォルトの : と同じ）
:[  → 角括弧インデントブロック（配列）
:{  → 波括弧インデントブロック（オブジェクト）
```

ただし `:{` は現在 `:` + `{` として「エントリー演算子の右辺にオブジェクトリテラル」と解釈される。
`:[` も同様に「エントリー演算子の右辺に配列リテラル」と解釈される。

→ **これらはfactor位置でのみ新しい意味を持つ**ため、中置演算子 `:` の右辺として `{` や `[` が来る既存コードとは競合しない（中置演算子が先に消費されるため）。

## 15. インデント検出の具体的アルゴリズム

### 前処理方式での疑似コード

```
function preprocessIndentBlocks(source):
    lines = source.splitLines()
    result = []
    indentStack = [0]  // 基底インデントレベル
    
    for line in lines:
        currentIndent = countLeadingSpaces(line)
        content = line.trimStart()
        
        if content == "":
            result.append("")  // 空行はそのまま
            continue
        
        // デインデント検出
        while currentIndent < indentStack.top():
            indentStack.pop()
            result.append(")")  // 閉じ括弧を挿入
        
        // 前の行がコロンで終わっていた場合
        if previousLineEndedWithColon:
            if currentIndent > indentStack.top():
                indentStack.push(currentIndent)
                // コロンの位置に応じた括弧を挿入
                result.append("(")
                previousLineEndedWithColon = false
            else:
                // インデントが増えなかった → エラーまたはコロンを通常演算子として扱う
                error("Expected indented block after ':'")
        
        result.append(content)
        
        // 行末のコロンを検出
        if content.endsWith(":") and not content.endsWith("::") and not content.endsWith("?:") and not content.endsWith("!:") and not content.endsWith(":="):
            previousLineEndedWithColon = true
            result.last() = result.last().dropLast(1)  // コロンを除去
    
    // 残りのデインデント
    while indentStack.size > 1:
        indentStack.pop()
        result.append(")")
    
    return result.join("\n")
```

**注意:** この疑似コードは単純化しており、コメントや文字列内のコロン、複合演算子 `::`, `?:`, `!:`, `:=` の判別が必要。

## 16. 「`:` は頭に来ないという前提」の崩壊の具体的影響

Issue #177の最新コメントで指摘されている：
> `:` を頭に持つインデント括弧が定義できればいい。ただ、 `:` は頭に来ないという前提が崩壊する

この「前提」が崩壊する具体的な箇所を調査する：

### 16.1 パーサーレベル

`assignmentOperator` で `:` は `-':' * !'=' * !':' ` として定義されている。
これは**中置演算子**として定義されているため、左辺が必ず存在する。

`:` をfactorに追加しても、factorは中置演算子よりも優先的にパースされるため、パーサーレベルでは直接的な衝突はない。

ただし、以下のような構造では注意が必要：

```xarpite
a + :
    b
    c
```

ここで `+` の右辺として `:` インデントブロックがfactorとしてパースされる。
これは `a + (b; c)` と等価になる。**問題なし。**

### 16.2 意味論レベル

`:` がfactorとして「括弧」を表す場合、以下の場所でインデント括弧が使用可能になる：

1. **式の先頭**: `:\n  body` → `(body)`
2. **中置演算子の右辺**: `a + :\n  body` → `a + (body)`
3. **前置演算子の後**: `!:\n  body` → `!(body)` — **注意:** `!:` はラベル定義演算子
4. **括弧内**: `f(:\n  body)` → `f((body))`

**3番の問題:**
`!:` は `executionOperator` で消費されるため、`!` + `:` (factor) とは区別される。
`executionOperator` は `stream` レベルで処理されるため、factor位置の `:` とは階層が異なる。

ただし、以下のケース：
```xarpite
!:
    body
```

`!` が前置演算子（`unaryExclamation`）として消費される。その後、factorとして `:` が試行される。
**`!` の後のfactorとして `:` インデントブロックがマッチする。**
→ 結果: `!(body)`

一方、`label !: body` は `stream` レベルの `executionOperator` で処理される。
```xarpite
label !:
    body
```
→ `executionOperator` で `!:` が消費され、右辺に `body` が来る。**インデント括弧ではない。**

**区別は可能。** `!:` は `executionOperator` で先に消費されるため。

### 16.3 エディタ・ツールレベル

現在のシンタックスハイライトやリンター等が `:` を常に演算子として扱っている場合、factor位置の `:` は新しいカテゴリとして認識させる必要がある。

## 18. xarpegパーサーライブラリの制約分析

### 18.1 xarpeg 5.1.0の能力

xarpegはPEGベースのパーサーコンビネータライブラリで、以下の機能を持つ：

- **DSL構成**: Kotlin演算子オーバーロードによる直感的なパーサー定義
- **再帰サポート**: `ref { }` による遅延参照
- **位置追跡**: `mapEx { context, result -> }` による位置情報の取得
- **テキストキャプチャ**: `+` 演算子と `result.text(context)` による原文取得
- **左結合演算子**: `leftAssociative()` による適切な優先順位処理
- **順序付き選択**: PEGの `or()` による決定的な選択

### 18.2 xarpegの制約

**致命的な制約: 状態管理が不可能**

- `Parser<T>` 型にはユーザー定義の状態パラメータを持つ仕組みがない
- `mapEx { context, result }` の `context` は読み取り専用
- パーサー間で共有可能なミュータブル状態が存在しない
- 文法定義はクラスフィールドとして定義され、実行時の動的変更は不可

**結果として:**
- xarpegパーサー内部でインデントレベルを追跡することは**不可能**
- インデント感知構文を実装するには、**前処理（プリプロセス）方式**を採用するしかない

### 18.3 パース呼び出しフロー

```
eval(src) → withEvaluator() → evaluator.get(src) →
XarpiteGrammar(location).rootParser.parseAllOrThrow(src) →
ParseResult<Node> (AST)
```

前処理を挿入する場合、`parseAllOrThrow(src)` の前に `src` を変換する。

## 19. 前処理方式の詳細設計

### 19.1 前処理の挿入ポイント

```kotlin
// 現在
val parseResult = XarpiteGrammar(location).rootParser.parseAllOrThrow(src)

// 変更後
val preprocessed = preprocessIndentBlocks(src)
val parseResult = XarpiteGrammar(location).rootParser.parseAllOrThrow(preprocessed)
```

### 19.2 コロン検出の精密な条件

前処理で「`:` がインデント括弧のマーカーであるかどうか」を判定する条件：

1. `:` の直後に水平空白（スペース・タブ）のみがあり、その後に改行がある
2. `:` の直前が `!`, `?`, `=`, `:` のいずれでもない（`!:`, `?:`, `:=`, `::` を除外）
3. 次の非空行のインデントレベルが現在の行よりも深い
4. `:` が文字列リテラル・コメント・正規表現の内部にない

### 19.3 文字列リテラル・コメント内のコロン検出回避

前処理段階で以下を認識してスキップする必要がある：

| 構文 | 開始 | 終了 | エスケープ |
|------|------|------|-----------|
| 生文字列 | `'` | `'` | `''` で `'` を表現 |
| テンプレート文字列 | `"` | `"` | `\"` でエスケープ |
| 埋め込み文字列 | `%>` | `<%` | `<%%` でエスケープ |
| 行コメント | `#` or `//` | 改行 | なし |
| ブロックコメント | `/*` | `*/` | ネスト可能 |
| 正規表現 | `/` | `/` | `\/` でエスケープ |
| クォートID | `` ` `` | `` ` `` | `\` でエスケープ |

**これは非常に複雑であり、前処理段階でパーサーの一部を再実装することになる。**

### 19.4 前処理の複雑さの見積もり

前処理で必要な処理：
1. **字句解析レベル**: 文字列・コメント・正規表現の境界を認識する簡易レキサー
2. **インデントレベル追跡**: 各行のインデント深さを計算
3. **コロン検出**: 上記条件を満たす `:` を特定
4. **括弧挿入**: 適切な位置に `(` と `)` を挿入
5. **位置マッピング**: エラーメッセージの行番号を元のソースに対応させる

**見積もり: 200〜400行のKotlinコード**

### 19.5 位置マッピングの問題

前処理でテキストを変換すると、パーサーが報告するエラー位置が元のソースコードの位置と一致しなくなる。

**解決策:**
1. 括弧文字を挿入する代わりに、元の `:` の位置に特殊な不可視文字を置く → パーサーがそれを認識
2. 位置のオフセットテーブルを維持し、エラー報告時に逆変換する
3. 括弧を挿入せず、改行をセミコロンに変換するだけにする（位置ずれが小さい）

## 20. テストケースへの具体的影響シミュレーション

### 20.1 影響を受けるパターンの分類

テストコード全体で `:` が行末に来るケースは**約227箇所**。これらを分類すると：

| パターン | 件数 | 影響 |
|---------|------|------|
| 三項演算子チェーン `:` 改行 | ~90件 | 前処理でインデント括弧と誤認される可能性 |
| オブジェクトリテラル内の `:` | ~50件 | `{a: \n 1}` が `{a: (1)}` に変換される |
| メソッド定義内の `:` | ~20件 | `method: this -> ...` が影響 |
| 名前付きパラメータ `:` | ~40件 | `[indent: "  "]` が影響 |
| メソッドチェーン `::` と `:` | ~15件 | `::` は除外されるが隣接する `:` に注意 |

### 20.2 三項演算子の問題の深掘り

三項演算子チェーンの典型例：
```xarpite
method == "apple"  ? (() -> "Fallback apple") :
method == "banana" ? (...) :
method == "cherry" ? (...) :
                     NULL
```

ここで `:` は三項演算子の一部であり、改行後にインデントされた次の `method` や `NULL` は三項演算子の右辺（else節）である。

**前処理方式での判定:**
- `:` の後に改行があり、次の行がインデントされている → インデント括弧？
- しかし、`?` の後に来る `:` は三項演算子の一部

**判別方法:**
1. **前処理段階では三項演算子かどうか判別不可能** — 前処理は字句レベルの処理であり、構文構造を理解していない
2. **結論: 前処理方式では三項演算子の `:` とインデント括弧の `:` を区別できない**

### 20.3 前処理方式の根本的な限界

前処理で「この `:` はインデント括弧の開始か？」を判定するには、構文解析レベルの文脈が必要：

- `:` が `condition` パーサー内の三項演算子の一部かどうか → **構文解析が必要**
- `:` が `assignmentOperator` のエントリー演算子かどうか → **構文解析が必要**
- `:` がインデント括弧のマーカーかどうか → **構文解析が必要**

→ **前処理方式では、`:` をインデント括弧のマーカーとして使うことは技術的に困難。**

### 20.4 再考: `:` 以外のマーカーの必要性

上記の限界を踏まえると、インデント括弧のマーカーとして `:` を使うことには深刻な問題がある。代替案：

| マーカー | 既存の使用 | 衝突リスク |
|---------|-----------|-----------|
| `\|` (行末パイプ) | パイプ演算子 `\|` | 低（パイプは通常右辺がある） |
| `->:` (矢印コロン) | `->` + `:` | 中（前処理で検出可能） |
| `=>:` (太矢印コロン) | `=>` + `:` | 中 |
| `\:` (バックスラッシュコロン) | `\` は前置演算子 | 低 |

**しかし、いずれの場合も前処理段階で文字列内部かどうかの判定が必要。**

## 21. パーサー内部方式の再検討

### 21.1 xarpegの制約を迂回する方法

xarpeg自体は状態管理をサポートしないが、**Grammarクラスのフィールドとして状態を保持する**ことは可能かもしれない：

```kotlin
class XarpiteGrammar(val location: String) {
    // 状態として現在のインデントスタックを保持
    private val indentStack = mutableListOf(0)
    
    val colonIndentBlock: Parser<Node> = -':' * customBrParser * ref { expression } * customDedentParser map { ... }
}
```

ただし、**PEGパーサーはバックトラックする**ため、ミュータブル状態の管理は非常に危険：

1. パーサーが `:` をインデント括弧として試行する
2. マッチに失敗してバックトラックする
3. 状態（indentStack）が変更されたまま → **状態が壊れる**

**バックトラック時の状態復元が必要**だが、xarpegにはそのフックが存在しない。

### 21.2 バックトラック安全な状態管理

PEGパーサーでバックトラック安全に状態を管理する方法：

1. **不変の状態**: 各パーサーが状態のスナップショットを受け取り、新しい状態を返す
   - xarpegのParser<T>はTのみを返すため、`Parser<Pair<Node, IndentState>>` のようにする必要がある
   - しかし、既存のすべてのパーサーの型が変わるため非現実的

2. **グローバルなバックトラック追跡**: パース位置と状態の対応表を管理
   - xarpegがパース位置を公開していれば可能だが、実装の詳細に依存

3. **副作用のないインデント検出**: テキスト位置からインデントレベルを計算する純粋関数
   - 「改行の直後の空白数」はテキスト自体から計算可能
   - パーサーがテキスト位置を知っていれば、状態なしでインデントレベルを推定できる

### 21.3 方式: テキスト位置ベースのインデント推定

`mapEx { context, result -> }` で `result.start` （パース位置）が取得できる。これを使えば：

```kotlin
val colonIndentBlock: Parser<Node> = (-':').mapEx { context, result ->
    val pos = result.start
    val text = ... // context からテキスト全体を取得
    val colonIndent = computeIndentAt(text, pos)
    // 次の行のインデントが colonIndent より深いことを確認
    // 深いインデントの行をすべてブロック内容として切り出す
    // ...
}
```

**問題:** xarpegの `context` からテキスト全体を取得できるかどうかが不明。`result.text(context)` は「マッチした部分のテキスト」を返すが、テキスト全体を返すかどうかは不明。

**仮にテキスト全体にアクセスできたとしても:**
- パーサー内で「次の行のインデントレベル」を判定するロジックが必要
- これは実質的に前処理と同じ処理をパーサー内部で行うことになる
- PEGの特性上、この処理は何度もバックトラックにより繰り返される可能性がある

4. 三項演算子との区別が構文解析レベルでのみ可能

**利益が限定的な理由:**
1. Xarpiteは比較的コンパクトなスクリプトで使用される（巨大プログラムは想定外）
2. `{}` がコードブロックではないため、インデント括弧が最も役立つ場面（if/else/while/for）が存在しない
3. 括弧の種類（`()`, `[]`, `{}`）の区別がインデント構文では表現しにくい

## 23. 代替案の総合比較

### 23.1 案一覧

| # | 構文 | 例 | 衝突リスク | 実装難度 |

| 1 | 行末 `:` + インデント | `-> :\n  body` | 高（三項演算子と衝突） | 高 |
| 2 | 行末 `\|` + インデント | `-> \|\n  body` | 低 | 高 |

### 23.2 推奨案の再評価

**第20節の分析を踏まえた結論:**

パーサー内部方式で `:` をfactor位置に追加する案（案6）が最も有望。

**案6の詳細:**
- `:` がfactor位置に来た場合のみインデント括弧として機能
- 中置演算子の `:` は左辺が存在するため衝突しない
- 三項演算子の `:` は `condition` パーサー内で消費されるため衝突しない
- ただし、xarpegでインデント検出を行う技術的な課題が残る

### 23.3 案6の実現に向けた具体的ステップ

1. **xarpeg拡張の調査**: `mapEx` の `context` パラメータからテキスト全体にアクセスできるか確認

3. **バックトラック安全性の検証**: PEGのバックトラックで状態が壊れないか確認
4. **テスト**: 既存の337テストがすべて通過することを確認
5. **エラーメッセージ**: インデントエラーの適切な報告
6. **ドキュメント**: 新構文の仕様と使用例の文書化

## 24. 「Issueコメントの `->:` の例」のパースフロー詳細トレース

Issue #177の最新コメント：
```xarpite
f := args ->:
    a
    b
```

### 24.1 現在のパーサーでのパースフロー（仮にインデント括弧が実装された場合）

```
入力: "f := args ->:\n    a\n    b"

1. rootParser = -b * (expression * -b).optional
2. expression = semicolons
3. semicolons → stream (行頭)
4. stream → commas * assignmentOperatorPart * ref { stream }
5. commas → ... → factor → identifier("f")
6. assignmentOperatorPart → ":=" (代入演算子)
7. ref { stream } → 右辺のストリーム
8. 右辺: commas → ... → factor → identifier("args")
9. assignmentOperatorPart → "->" (代入演算子)
10. ref { stream } → 右辺のストリーム
11. 右辺: commas → ... → factor → ??
```

ステップ11で、factorとして `:` が試行される。

現在のfactor:
```
jump + hexadecimal + identifier + quotedIdentifier + float + integer + rawString + templateString + embeddedString + regex + brackets
```

`:` はこれらのどれにもマッチしない。

**インデント括弧を追加した場合:**
```
jump + hexadecimal + identifier + quotedIdentifier + float + integer + rawString + templateString + embeddedString + regex + brackets + colonIndentBlock
```

`colonIndentBlock` パーサーが `:` + 改行 + インデント増加をマッチし、ブロック内の式をパースする。

```
12. colonIndentBlock → ':'改行 → インデントレベル確認 → expression("a\n    b") → SemicolonsNode([a, b])
13. 結果: BracketsLiteralSimpleRoundNode(SemicolonsNode([a, b]))
```

**最終AST:**
```
InfixColonEqualNode(
    "f",
    InfixMinusGreaterNode(
        "args",
        BracketsLiteralSimpleRoundNode(
            SemicolonsNode(["a", "b"])
        )
    )
)
```

これは `f := args -> (a; b)` と等価。**正しい。**

### 24.2 `colonIndentBlock` パーサーの実装イメージ

```kotlin
val colonIndentBlock: Parser<Node> = (-':').result.mapEx { context, colonResult ->
    // コロンの位置を取得
    val colonPos = colonResult.start
    
    // ここでインデント検出を行う必要がある
    // しかし、xarpegの context からテキスト全体にアクセスする方法が不明
    // → xarpegの拡張が必要かもしれない
    
    // 仮にテキストにアクセスできたとして：
    // 1. コロンの後に改行があるか確認
    // 2. 次の非空行のインデントレベルを計算
    // 3. そのインデントレベル以上の連続行をブロックとして切り出す
    // 4. 切り出したブロックをexpressionとしてパース
}
```

**根本的な問題:** PEGパーサーの `map` / `mapEx` 内でサブパーサーを呼び出すことは通常できない。パーサーの結果を変換するだけであり、新しいパースを開始する場所ではない。

→ **`colonIndentBlock` はパーサーコンビネータとして正しく構成する必要がある。**

つまり:
```kotlin
val colonIndentBlock: Parser<Node> = -':' * -br * indentedBlock
```

ここで `indentedBlock` は「現在の行よりインデントが深い連続行」をパースするパーサー。

しかし、`indentedBlock` はインデントレベルを知る必要がある → **状態が必要** → xarpegの制約に抵触。

## 25. xarpegの拡張可能性

### 25.1 xarpegのソースコードへのアクセス

xarpegは `io.github.mirrgieriana:xarpeg:5.1.0` としてMaven Centralに公開されている。
リポジトリオーナーは同じ（MirrgieRiana）であるため、xarpeg自体の改修も選択肢に入る。

### 25.2 xarpegに必要な拡張

インデント感知パーサーを実現するために必要なxarpegの拡張：

1. **状態付きパーサー**: `Parser<T>` を `Parser<S, T>` に拡張し、状態 `S` を持てるようにする
   - バックトラック時の状態復元を自動的に行う
   - 非常に大きなAPI変更であり、既存コードの大幅な書き換えが必要

2. **条件付きパーサー**: パース位置のテキスト内容に基づいて動的にパースを制御する
   - `when(predicate: (String, Int) -> Boolean)` のようなコンビネータ
   - 入力テキストと現在位置を受け取り、パース続行の可否を判定

3. **カスタムパーサー**: ユーザーが生のパースロジックを実装できるAPI
   - `Parser.create { input, pos -> ParseResult<T>? }` のようなファクトリ
   - これが最も柔軟で、最小限のAPI変更で済む

### 25.3 カスタムパーサーによる実装

xarpegに `Parser.create` のようなAPIがあれば：

```kotlin
val colonIndentBlock: Parser<Node> = Parser.create { input, pos ->
    // pos の位置に ':' があるか確認
    if (input[pos] != ':') return@create null
    
    // ':' の後に改行があるか確認
    val afterColon = pos + 1
    val newlinePos = input.indexOf('\n', afterColon)
    if (newlinePos < 0) return@create null
    val between = input.substring(afterColon, newlinePos)
    if (between.any { it != ' ' && it != '\t' }) return@create null
    
    // 次の行のインデントレベルを計算
    val colonLineStart = input.lastIndexOf('\n', pos - 1) + 1
    val colonIndent = countIndent(input, colonLineStart)
    val blockIndent = countIndent(input, newlinePos + 1)
    if (blockIndent <= colonIndent) return@create null
    
    // blockIndent 以上のインデントの連続行をブロックとして切り出す
    val blockEnd = findBlockEnd(input, newlinePos + 1, blockIndent)
    val blockContent = input.substring(newlinePos + 1, blockEnd)
    
    // ブロック内容をexpressionとしてパース（再帰的パース）
    val innerResult = expression.parse(blockContent, 0)
    if (innerResult == null) return@create null
    
    ParseResult(
        BracketsLiteralSimpleRoundNode(innerResult.value, Position(location, pos)),
        pos, blockEnd
    )
}
```

**問題:** `expression` の再帰呼び出しが必要だが、内部テキストの位置オフセットの管理が複雑。

## 26. 最も現実的な実装パス

以上の考察を総合すると、最も現実的な実装パスは以下の通り：

### パス1: 前処理 + 新マーカー（推奨度: ★★★）

`:` の代わりに `\:` や `|:` などの**新しい2文字トークン**をインデント括弧のマーカーとして導入し、前処理で括弧に変換する。

**利点:**
- 既存の `:` 使用との衝突がゼロ
- 前処理で正確に検出可能（2文字トークンは誤検出が少ない）
- xarpegの改修が不要
- エラーメッセージの品質は低いが、機能的には正しい

**欠点:**
- Python方式の直感的な `:` が使えない
- 新しい記号を覚える必要がある

### パス2: xarpeg にカスタムパーサーAPIを追加（推奨度: ★★☆）

xarpeg 5.2.0 として `Parser.create` APIを追加し、colonIndentBlock パーサーを実装する。

**利点:**
- `:` をそのままインデント括弧マーカーとして使える
- パーサー内部で正確なインデント検出が可能
- エラーメッセージの品質が高い

**欠点:**
- xarpegライブラリ自体の改修が必要
- カスタムパーサー内のバグがデバッグしにくい
- PEGバックトラックとの相互作用の検証が必要

### パス3: 前処理 + `:` + ヒューリスティック（推奨度: ★☆☆）

前処理段階で `:` + 改行をヒューリスティックに検出し括弧に変換する。

**利点:**
- Python方式の `:` が使える
- xarpegの改修が不要

**欠点:**
- 三項演算子との衝突を完全には回避できない
- ヒューリスティックの誤検出・見逃しのリスク
- 字句解析の再実装が必要

## 27. 結論

### 27.1 技術的結論

1. **`:` をインデント括弧マーカーとして使うことは、三項演算子 `? :` との区別が前処理段階では不可能であるため困難。**
2. **パーサー内部方式（`:` をfactor位置に追加）は理論的には可能だが、xarpegの現在のAPIではインデント検出が実装できない。**
3. **最も現実的なのは、新しい2文字トークンをマーカーとして使う前処理方式。**

### 27.2 設計上の推奨

- **第一候補**: `|:` をインデント括弧マーカーとして使う（Yamlのブロックスカラー記法からの類推）
- **第二候補**: xarpegを拡張して `:` をfactor位置でインデント括弧として使えるようにする
- **長期的**: xarpegにインデント感知のファーストクラスサポートを追加する

### 27.3 Issue #177のコメントへのフィードバック

> `:` を頭に持つインデント括弧が定義できればいい。ただ、 `:` は頭に来ないという前提が崩壊する

**分析結果:** `:` がfactor位置に来てもパーサーの優先順位により中置演算子の `:` とは自然に区別可能。「前提の崩壊」は文法レベルでは問題にならない。ただし、**インデント検出の実装**がxarpegの制約により困難。xarpeg自体の拡張が必要。

## 28. 3種の括弧のインデント括弧としての使い分け設計

### 28.1 括弧の意味論の整理

Xarpiteの3種の括弧は、リテラル（独立）形式と後置（レシーバ付き）形式で異なる意味を持つ：

| 括弧 | リテラル形式 | 後置形式 |
|------|-------------|---------|
| `()` | 新しい変数スコープを作成し式を評価 | 関数の即時呼び出し |
| `[]` | 配列を作成（ストリーム展開あり） | 関数の部分適用（引数バインド） |
| `{}` | オブジェクトを作成（キー値ペア） | オブジェクト継承（親の設定） |

### 28.2 インデント括弧で最も需要がある括弧種類

ユーザーがインデント括弧を使いたいと思う場面を考える：

**1. コードブロック（関数本体、制御構造）→ `()` が必要**
```xarpite
# 現在
f := args -> (
    result := compute(args)
    result + 1
)

# インデント括弧で
f := args ->:
    result := compute(args)
    result + 1
```

**2. 配列リテラル → `[]` が必要**
```xarpite
# 現在
items := [
    "apple"
    "banana"
    "cherry"
]

# インデント括弧で
items := :[
    "apple"
    "banana"
    "cherry"
```

**3. オブジェクトリテラル → `{}` が必要**
```xarpite
# 現在
config := {
    host: "localhost"
    port: 8080
    debug: TRUE
}

# インデント括弧で
config := :{
    host: "localhost"
    port: 8080
    debug: TRUE
```

**4. クロージャ渡し → `()` + `=>` が必要**
```xarpite
# 現在
LAUNCH(task => (
    OUT << "Hello"
    OUT << "World"
))

# インデント括弧で
LAUNCH(task =>:
    OUT << "Hello"
    OUT << "World"
```

### 28.3 需要の優先順位

1. **`()` コードブロック** — 最も頻繁。関数本体、ラムダ、制御構造の本体
2. **`{}` オブジェクトリテラル** — 中程度。複雑なオブジェクト定義
3. **`[]` 配列リテラル** — 低め。配列は通常コンパクト

→ デフォルトの `:` は `()` （コードブロック/スコープ）として扱うのが最も自然。

### 28.4 括弧種類の指定構文案

**案1: `:` のみ（常に `()`）**
```
:   → ()
```
- 最もシンプル
- オブジェクトや配列のインデント構文が使えない
- Issue #177のユースケースはほぼカバー

**案2: `:(`, `:[`, `:{`**
```
:(  → ()   （デフォルトの : と同じ）
:[  → []   （配列）
:{  → {}   （オブジェクト）
```
- 括弧種類を明示的に指定可能
- `:{` は現在 `:` + `{` として合法だが、factor位置でのみ新解釈
- 閉じ括弧不要のメリットを維持

**案3: 開き括弧 + `:` + インデント（Issue本文の案B）**
```
( => :   → ( => )   （ラムダ丸括弧）
[ :      → [ ]      （配列）
{ :      → { }      （オブジェクト）
```
- 開き括弧を明示するので括弧種類が自明
- `:` は「閉じ括弧をインデントで代用」というマーカー
- 開き括弧があるので閉じ括弧省略のメリットが半減

**推奨: 案2**

理由：
- 案1は用途が限定的すぎる
- 案3は開き括弧を書く手間があり、インデント括弧のメリットが薄い
- 案2は完全に新しいトークン群（`:(`, `:[`, `:{`）として導入でき、既存構文との衝突リスクが低い

### 28.5 案2の詳細な文法定義

```
indentBlock    ::= colonParen | colonBracket | colonBrace
colonParen     ::= ':(' NEWLINE INDENT expression DEDENT   → BracketsLiteralSimpleRoundNode
colonBracket   ::= ':[' NEWLINE INDENT expression DEDENT   → BracketsLiteralSimpleSquareNode
colonBrace     ::= ':{' NEWLINE INDENT expression DEDENT   → BracketsLiteralSimpleCurlyNode

# Arrowed variants
colonArrowParen   ::= ':(' commas '=>' NEWLINE INDENT expression DEDENT  → BracketsLiteralArrowedRoundNode
colonArrowBracket ::= ':[' commas '=>' NEWLINE INDENT expression DEDENT  → BracketsLiteralArrowedSquareNode
# colonArrowBrace は不要（現在も arrowedCurly は不使用/エラー）

factor ::= ... | indentBlock
```

**注意:** `:(` は2文字トークンとしてパースする必要がある。
現在 `:` は `assignmentOperator` で中置演算子として定義されており、`(` は `brackets` でfactorとして定義されている。
`:(` がfactor位置に来た場合、`:` が先に中置演算子として消費されるか、`:(` がfactorとしてマッチするかは、パーサーの評価順序に依存する。

**PEGの特性:**
factor位置では中置演算子は試行されない（中置演算子は左辺の後にのみ出現する）。
したがって、factor位置での `:(` は中置演算子 `:` とは衝突せず、新しいfactorとして正しくマッチする。

## 29. エッジケースの網羅的列挙

### 29.1 空のインデントブロック

```xarpite
result := :
```

`:` の後にインデントされた行がない場合。
→ **エラーにするべき。** 空のブロックは `()` (NULL) で書くべき。

### 29.2 インデントブロック内のさらなるインデント括弧

```xarpite
f := args ->:
    inner := x ->:
        x + 1
    inner(args)
```

ネストされたインデント括弧。
→ インデントスタックにより正しく処理可能。外側のブロックは4スペース、内側は8スペース。

### 29.3 インデントブロックとセミコロンの混在

```xarpite
f := :
    a := 1; b := 2
    a + b
```

インデントブロック内でセミコロンによる文の区切り。
→ **問題なし。** 既存の `semicolons` パーサーが処理。

### 29.4 インデントブロック後の継続式

```xarpite
result := (:
    compute()
) + 10
```

案2では `:(` の後にインデントブロックがあり、その後に `+ 10` がある。
→ ブロックの終了（デインデント）で自動的に `)` が挿入されるため、`+ 10` は外側の式の一部として正しくパースされる。

ただし、閉じ括弧が省略されるため：
```xarpite
result := :(
    compute()
+ 10
```
→ デインデントで `)` が挿入され、`+ 10` が外側の式に。**正しい。**

### 29.5 タブとスペースの混在

```xarpite
f := :
    body_with_spaces
	body_with_tab
```

タブとスペースのインデントが混在する場合。
→ **Pythonと同様に、タブとスペースの混在はエラーにするべき。**
または、タブを一定数のスペースに変換する。

### 29.6 コメントのみの行

```xarpite
f := :
    # これはコメント
    body
```

コメント行のインデントはブロックの判定に影響するか？
→ **コメント行はインデント判定から除外するべき。** 空行と同様に扱う。

### 29.7 空行

```xarpite
f := :
    a := 1

    a + 1
```

空行がブロック内に含まれる場合。
→ **空行はインデント判定から除外。** ブロックの終了は非空行のデインデントで判定。

### 29.8 インデント括弧を後置位置で使う

```xarpite
obj := MyClass:{
    a: 1
    b: 2
```

`MyClass` の後に `:{` が来て、オブジェクト継承 + インデント括弧。
→ `:{` がfactor位置の新トークンなら、これは `MyClass` の後のfactorとして解釈され、後置括弧（`BracketsRightSimpleCurlyNode`）ではなくリテラル括弧になってしまう。

**問題:** 後置位置での `:{` をサポートするには、`rightOperator` にもインデント括弧を追加する必要がある。

```kotlin
val rightOperator: Parser<(Node) -> Node> = or(
    // ... 既存の後置括弧 ...
    -s * colonIndentBlockRight,  // 新規: 後置インデント括弧
)
```

### 29.9 三項演算子の直後のインデント括弧

```xarpite
result := condition ? :
    true_branch
: false_branch
```

三項演算子の `?` の後に `:` インデントブロック。
→ 三項演算子は `condition` パーサーで処理される。`?` の後に来る式（真の枝）としてfactorが評価され、`:` インデントブロックがマッチする。偽の枝の `:` は三項演算子の一部として消費される。
→ **正しく動作する可能性がある**が、偽の枝の `:` がインデント括弧と誤認されないか確認が必要。

### 29.10 パイプ演算子との組み合わせ

```xarpite
1 .. 10 | x =>:
    result := x * 2
    OUT << result
    result
```

パイプ演算子の右辺にクロージャ + インデント括弧。
→ `|` の右辺で `x =>` が消費され、その後のfactorとして `:` インデントブロックがマッチ。
→ **正しく動作するはず。**

### 29.11 複数のインデントブロックが連続する場合

```xarpite
if := condition, then, else_ ->
    condition ? then() : else_()

if(x > 0; :
    OUT << "positive"
; :
    OUT << "negative"
)
```

引数としてインデント括弧を複数渡す場合。
→ `;` で区切られた各引数がfactorとして評価され、それぞれ `:` インデントブロックがマッチ。
→ **インデントの管理が複雑になる。** 第2引数のインデントブロックのインデントレベルはどうなるか？

これは実装上の大きな課題。各引数のインデントブロックが同じインデントレベルにあると、デインデントの判定が困難になる。

### 29.12 行末の空白がある場合

```xarpite
f := :   
    body
```

`:` の後にスペースがあり、その後に改行。
→ 水平空白はスキップして改行を検出する。**問題なし。**

### 29.13 `:` の直後にコメントがある場合

```xarpite
f := : # この関数は...
    body
```

`:` の後にコメント、その後に改行。
→ コメントはスキップして改行を検出する。**問題なし。**

### 29.14 インデントブロックの最後の式が値を返す場合

```xarpite
result := :
    a := compute()
    a + 1
```

インデントブロック ≡ `(a := compute(); a + 1)` なので、`a + 1` の値が返る。
→ **問題なし。** `semicolons` パーサーの最後の式が値になる。

### 29.15 文字列内のコロン

```xarpite
msg := "key: value"
```

文字列内の `:` はインデント括弧として解釈されてはならない。
→ 前処理方式では文字列の境界を認識する必要がある。パーサー内部方式では文字列パーサーが先にマッチするため問題なし。

## 30. 後置インデント括弧の設計

### 30.1 後置位置でのインデント括弧の必要性

Issue #177の最も重要なユースケースは**関数呼び出し時のクロージャ渡し**：

```xarpite
LAUNCH ( => :
    OUT << "Hello, World!"
```

これは後置括弧 `LAUNCH(...)` のクロージャバリエーションであり、リテラルfactorではない。
後置位置でのインデント括弧が必要。

### 30.2 後置インデント括弧の構文

`rightOperator` に追加する後置インデント括弧：

```
# 単純な後置インデント括弧
f:(         → f( ... )         （関数呼び出し + インデントブロック）
f:[         → f[ ... ]         （部分適用 + インデントブロック）
f:{         → f{ ... }         （オブジェクト継承 + インデントブロック）

# Arrow付き後置インデント括弧
f:( args => → f( args => ... ) （クロージャ呼び出し + インデントブロック）
f:[ args => → f[ args => ... ] （クロージャ部分適用 + インデントブロック）
```

### 30.3 Issue本文の構文との対応

Issue #177の案B:
```xarpite
LAUNCH ( => :
    OUT << "Hello, World!"
```

これは「`(` で開始 → `=>` でクロージャ → `:` でインデントブロック開始 → 閉じ括弧省略」。

しかし、この構文は `LAUNCH` + 後置 `(` + `=>` + `:` という4要素の組み合わせであり、パーサーの設計が複雑になる。

**簡略化案:**
```xarpite
LAUNCH :( =>
    OUT << "Hello, World!"
```

`:(` を「インデント丸括弧開始」として、`=> ` の後にインデントブロック本体が続く。
ただし、この場合 `:(` はレシーバ `LAUNCH` の直後に来るので後置演算子として機能する。

### 30.4 後置と前置の統一設計

**統一ルール: `:(`, `:[`, `:{` はコンテキストにより前置（factor）または後置（rightOperator）として機能する**

- 式の先頭 → リテラル括弧（前置）
- 式の直後（レシーバあり）→ 後置括弧

これは既存の `(`, `[`, `{` と同じ振る舞い：
- `(expr)` → リテラル丸括弧
- `f(args)` → 後置丸括弧（関数呼び出し）

`:(`, `:[`, `:{` も同様に振る舞えばよい。

```xarpite
process := data -> (
```

### 31.4 オブジェクト定義

```xarpite
Person := {
    new: name, age -> Person{

        age: age
    }
    greet: this -> "I'm $(this.name), $(this.age) years old"
    `&_`: this -> "Person[$(this.name)]"
}
```

**After:**
```xarpite

    new: name, age -> Person :{
        name: name
        age: age
    greet: this -> "I'm $(this.name), $(this.age) years old"
    `&_`: this -> "Person[$(this.name)]"
```

**問題:** ネストされたインデント括弧のデインデントが `new` の本体と `Person` の本体で正しく処理されるか？
`Person:{` の本体は4スペースインデント。`new` の値の中の `Person:{` は8スペースインデント。
`greet` の行は4スペースインデントなので、8スペースのブロックは終了し、4スペースのブロックに戻る。
→ **正しく動作する。**

### 31.5 制御構造的パターン

**Before:**
```xarpite
1 .. 100 | (
    x := _ * 2
    x > 50 ? (
        OUT << "Large: $x"
    ) : (
        OUT << "Small: $x"
    )
    x
)
```

**After:**
```xarpite
1 .. 100 | :(
    x := _ * 2
    x > 50 ? :(
        OUT << "Large: $x"
    : :(
        OUT << "Small: $x"
    x
```

**問題:** 三項演算子の偽の枝（`:`の後）でインデント括弧を使う場合、`:` がインデント括弧マーカーなのか三項演算子の一部なのか曖昧になる。

→ `:(` は2文字トークンなので、三項演算子の `:` + `(` とは区別可能…と思いきや、`: (` と `: :(` の区別が必要。

実は、三項演算子の `:` の後に来る式（偽の枝）はfactorから始まる。ここでfactorとして `:(` がマッチすれば、それはインデント括弧になる。

→ **三項演算子の偽の枝でもインデント括弧は使用可能。**

### 31.6 パイプチェーン

**After:**

パイプ `|` が4スペースインデント（外側）にあるため、`:(` のブロック（8スペースインデント）は正しくデインデントで閉じられる。
→ **正しく動作する。**

## 32. `:(` / `:[` / `:{` のパースにおける技術的詳細

### 32.1 `:{` の既存構文との衝突分析

現在のXarpiteで `:` の直後に `{` が来るケース：

```xarpite
key: {a: 1; b: 2}
```

これは `InfixColonNode(key, BracketsLiteralSimpleCurlyNode({a:1;b:2}))` — エントリー演算子の右辺にオブジェクトリテラル。

`:{` をインデント括弧として導入した場合、これは？

→ `key` がfactorとしてパースされた後、`:` が中置演算子（`assignmentOperator`）としてマッチする。右辺として `{a: 1; b: 2}` が通常のオブジェクトリテラルとしてパースされる。
→ `:{` が2文字トークンとしてfactorに追加されても、この位置では中置演算子 `:` が先にマッチするため、`:{` factorは試行されない。

**衝突なし。**

### 32.2 `:(` の既存構文との衝突分析

```xarpite
key: (expr)
```

→ 同上。中置演算子 `:` が先にマッチし、右辺に `(expr)` が来る。
→ `:(` factorは試行されない。

**衝突なし。**

### 32.3 factor位置での `:(` の動作

```xarpite
result := :(
    body
```

1. `result` → factor (identifier)
2. `:=` → assignmentOperator
3. 右辺のfactor: `:(` → colonIndentBlock → `BracketsLiteralSimpleRoundNode(body)`

**正しく動作する。**

### 32.4 後置位置での `:(` の動作

```xarpite
LAUNCH:(
    body
```

1. `LAUNCH` → factor (identifier)
2. rightOperator: `:(` → 後置インデント括弧 → `BracketsRightSimpleRoundNode(LAUNCH, body)`

**正しく動作する。**

ただし、これは `LAUNCH` と `:(` の間に空白がない場合。空白がある場合：

```xarpite
LAUNCH :(
    body
```

rightOperatorは `-s * ...` で始まるため、水平空白の後に `:(` がマッチ。
→ **正しく動作する。**

## 33. 総括: 推奨する完全な設計

### 33.1 構文

| 記法 | 意味 | 等価表現 |
|------|------|---------|
| `:(` + インデントブロック | 丸括弧（コードブロック/スコープ） | `( expression )` |
| `:[` + インデントブロック | 角括弧（配列） | `[ expression ]` |
| `:{` + インデントブロック | 波括弧（オブジェクト） | `{ expression }` |
| `:(` args `=>` + インデントブロック | Arrow付き丸括弧 | `( args => expression )` |
| `:[` args `=>` + インデントブロック | Arrow付き角括弧 | `[ args => expression ]` |

### 33.2 文法変更

```
# factorに追加
factor ::= ... | colonIndentLiteralBlock

# rightOperatorに追加
rightOperator ::= ... | colonIndentRightBlock

# 定義
colonIndentLiteralBlock ::= colonParen | colonBracket | colonBrace
  colonParen   ::= ':(' [commas '=>'] NEWLINE INDENT expression DEDENT
  colonBracket ::= ':[' [commas '=>'] NEWLINE INDENT expression DEDENT
  colonBrace   ::= ':{' NEWLINE INDENT expression DEDENT

colonIndentRightBlock ::= colonRightParen | colonRightBracket | colonRightBrace
  colonRightParen   ::= ':(' [commas '=>'] NEWLINE INDENT expression DEDENT
  colonRightBracket ::= ':[' [commas '=>'] NEWLINE INDENT expression DEDENT
  colonRightBrace   ::= ':{' NEWLINE INDENT expression DEDENT
```

### 33.3 ASTノード

既存のノード型をそのまま再利用。インデント括弧は対応する通常の括弧と同じノードに変換される：
- `:(` → `BracketsLiteralSimpleRoundNode` / `BracketsRightSimpleRoundNode`
- `:[` → `BracketsLiteralSimpleSquareNode` / `BracketsRightSimpleSquareNode`
- `:{` → `BracketsLiteralSimpleCurlyNode` / `BracketsRightSimpleCurlyNode`

→ **GetterCompiler, RunnerCompilerの変更は不要。**

### 33.4 実装方式

**推奨: 前処理方式**

`:(`, `:[`, `:{` は明確な2文字トークンであるため、前処理段階で正確に検出可能：
1. 文字列・コメント・正規表現の内部をスキップする簡易レキサー
2. `:(`, `:[`, `:{` を検出
3. その後の改行 + インデントブロックを検出
4. ブロックの終了位置（デインデント）を計算
5. 対応する閉じ括弧 `)`, `]`, `}` を挿入

**前処理で `:` 単独をマーカーとする場合の三項演算子との衝突問題は、`:(` / `:[` / `:{` を使うことで完全に回避される。**

### 33.5 残る技術課題

1. **前処理の字句解析**: 文字列・コメント・正規表現内の `:(` をスキップする処理
2. **インデントの正規化**: タブとスペースの混在への対応
3. **エラーメッセージ**: 前処理で挿入した括弧に起因するエラーメッセージの品質
4. **位置マッピング**: パーサーのエラー位置を元のソースに逆変換

### 33.6 段階的実装計画

**Phase 1: 最小実装**
- `:(` のみサポート（最も需要が高い丸括弧）
- 前処理方式で実装
- `:[`, `:{` は将来の拡張として保留

**Phase 2: 全括弧種類サポート**
- `:[`, `:{` を追加
- Arrow付きバリエーション（`:(args =>` 等）を追加

**Phase 3: エラーメッセージ改善**
- 位置マッピングの実装
- インデントエラーの親切なメッセージ

**Phase 4: xarpeg統合**
- xarpegにカスタムパーサーAPIを追加
- 前処理を廃止し、パーサー内部でインデント処理
