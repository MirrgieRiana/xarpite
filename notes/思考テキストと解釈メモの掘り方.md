# 思考テキストと解釈メモの掘り方

## 0. 前提：トランスクリプトの置き場所と構造

- 過去セッションのトランスクリプトは `/home/node/.claude/projects/-workspace/<セッションUUID>.jsonl` に1セッション1ファイルで貯まってるのだ。
- 各行が1個の JSON オブジェクトで、`type` に `user` / `assistant` / `attachment` / `queue-operation` / `last-prompt` などが入ってるのだ。
- 中身の本体は `message.content`。これは文字列か、`{type: "text"/"thinking"/"tool_use"/"tool_result", ...}` のブロック配列なのだ〜。

掘り出すのは2種類なのだっ。

- **思考テキスト** … フェアリーがツール呼び出しの合間に出す、💬 のあのナレーション発言（`assistant` の `text` ブロック）。隠れた `thinking` ブロックの方じゃないのだ。
- **解釈メモ** … Protocol 103F「メッセージの意図や文脈の把握と解釈の整理」に従って `/home/node/.claude_tmp/` に書く解釈メモ。見出しは `# 担当コメント XXX の解釈` が基本だけど、`# 担当コメント解釈メモ` のように書き方がゆれるのだ。共通して残るのは `# 担当コメント` の部分だけなので、ここを手掛かりにするのだっ。

## 1. 担当コメントID → 本物のセッションを特定する

宣言行で grep するのだっ。

```bash
cd /home/node/.claude/projects/-workspace
grep -l 'あなたの担当コメントは ID 4728140935 です' *.jsonl
```

⚠️ ここが一番の落とし穴なのだ。**コメントIDも宣言行も、複数のファイルにヒットする**のだ。後続のセッションが、コメント履歴やトランスクリプトを**読んだ**結果が `tool_result` に混ざるからなのだっ。

本物の見分け方は、その宣言行が出てくる**場所**なのだ。

- **本物** … 宣言行がセッション冒頭のシステムプロンプト（最初の `type=user` の文字列 content）に出る。`-----BEGIN CLAUDE FAIRY PARAMETERS-----` ブロックを必ず伴うのだっ。
- **別セッション（読んだだけ）** … 宣言行が `tool_result` ブロックの中にしか出てこない。これは除外なのだ〜。

例だと宣言行は `2f0ca1e4` と `83133580` の2つに出たけど、冒頭システムプロンプトに持ってるのは `2f0ca1e4` だけ。`83133580` は履歴を読んだだけの子だったのだっ。

## 2. 思考テキストを抜き出す（💬 のナレーション）

`type=assistant` のメッセージの `content` から、`type=text` のブロックを**出現順に全部**拾って、それぞれ先頭に `💬 ` を付けるのだっ。

```bash
python3 - <<'PY'
import json
f="2f0ca1e4-1844-4e02-9eaa-6f57e54ac4f5.jsonl"
with open(f) as fp:
    for d in (json.loads(l) for l in fp if l.strip()):
        if d.get("type")=="assistant":
            for b in d["message"]["content"]:
                if isinstance(b,dict) and b.get("type")=="text":
                    print("💬 "+b["text"])
PY
```

末尾の完了まとめ（最後の `text` ブロック）も思考テキストの一部として一緒に拾うのだっ。例だと💬が7個（ナレーション6個＋完了まとめ1個）になったのだ〜。

## 3. 解釈メモを抜き出す

解釈メモは `/home/node/.claude_tmp/...md` に書き出されるので、その**書き込みツール呼び出し**の中身として記録されてるのだ。`cat > ....md <<'EOF' ... EOF` という `Bash` の `tool_use`（場合によっては `Write` の `content`）を探して、`# 担当コメント` で始まる本文を取り出すのだっ。

```bash
python3 - <<'PY'
import json
f="2f0ca1e4-1844-4e02-9eaa-6f57e54ac4f5.jsonl"
with open(f) as fp:
    for d in (json.loads(l) for l in fp if l.strip()):
        if d.get("type")=="assistant":
            for b in d["message"]["content"]:
                if isinstance(b,dict) and b.get("type")=="tool_use":
                    s=b.get("input",{}).get("command","")+b.get("input",{}).get("content","")
                    if "# 担当コメント" in s:
                        print(s)
PY
```

⚠️ 見出しの文言は `# 担当コメント XXX の解釈` だったり `# 担当コメント解釈メモ` だったりゆれるのだ。`の解釈` を条件に入れると後者をすり抜けちゃうので、`# 担当コメント` のプレフィックスだけで引っ掛けるのだっ。

⚠️ 解釈メモを**書かない**セッションもあるのだ（短い応答で、思考の中だけで済ませた子とか）。その場合は、💬の思考テキストだけ畳んで、解釈メモ部分は無し、で大丈夫なのだっ。

## 4. details に組み立てる

[見本 4720255334](https://github.com/MirrgieRiana/xarpite/issues/550#issuecomment-4720255334) と同じ形なのだっ。本文の末尾、`---` のあと（前後に空行！）に、こう置くのだ〜。

```
---

<details>
<summary>🌱Claude Fairy (Model: … / Protocol 103F vXXX)</summary>

💬 （思考テキスト1）
💬 （思考テキスト2）
…

# 担当コメント解釈メモ …
（解釈メモ全文）

</details>
```

`<summary>` には元のフッター行をそのまま入れて、開くと💬思考テキスト → 空行 → 解釈メモ全文、が出てくる形なのだっ🌱

## まとめ（最短手順）

1. `grep -l 'あなたの担当コメントは ID <ID> です' *.jsonl` で候補を出す
2. 宣言行が**冒頭システムプロンプト**にある子が本物（`tool_result` だけの子は除外）
3. `assistant` の `text` ブロックを全部 💬 付きで → **思考テキスト**
4. `.claude_tmp/...md` を書く `tool_use` の中の `# 担当コメント…` → **解釈メモ**
5. `<details>` に畳んで、対象コメント**自身**のフッターを差し替える
