# 思考テキストと解釈メモの掘り方

過去セッションのトランスクリプトから、フェアリーの「思考テキスト」と「解釈メモ」を掘り出す手順をまとめます。

## 前提：トランスクリプトの置き場所と構造

過去セッションのトランスクリプトは `/home/node/.claude/projects/-workspace/<セッションUUID>.jsonl` に、1セッション1ファイルで蓄積されます。各行が1個の JSON オブジェクトで、`type` には `user`・`assistant`・`attachment`・`queue-operation`・`last-prompt` などが入ります。中身の本体は `message.content` で、文字列か、`{type: "text"/"thinking"/"tool_use"/"tool_result", ...}` のブロック配列です。

掘り出す対象は次の2種類です。

- 思考テキスト … フェアリーがツール呼び出しの合間に出す、ナレーション発言（`assistant` の `text` ブロック）です。隠れた `thinking` ブロックではありません。
- 解釈メモ … Protocol 103F「メッセージの意図や文脈の把握と解釈の整理」に従って `/home/node/.claude_tmp/` に書く解釈メモです。見出しは `# 担当コメント XXX の解釈` が基本ですが、`# 担当コメント解釈メモ` のように揺れます。共通して残るのは `# 担当コメント` の部分だけなので、ここを手掛かりにします。

## 担当コメント ID から本物のセッションを特定する

宣言行で grep します。

```bash
cd /home/node/.claude/projects/-workspace
grep -l 'あなたの担当コメントは ID 4728140935 です' *.jsonl
```

ここが最大の落とし穴です。コメント ID も宣言行も、複数のファイルにヒットします。後続のセッションが、コメント履歴やトランスクリプトを読んだ結果が `tool_result` に混ざるためです。

本物の見分け方は、その宣言行が出てくる場所です。

- 本物 … 宣言行がセッション冒頭のシステムプロンプト（最初の `type=user` の文字列 content）に出ます。`-----BEGIN CLAUDE FAIRY PARAMETERS-----` ブロックを必ず伴います。
- 別セッション（読んだだけ） … 宣言行が `tool_result` ブロックの中にしか出てきません。これは除外します。

たとえば、宣言行が `2f0ca1e4` と `83133580` の2つに出たとき、冒頭システムプロンプトに持っているのは `2f0ca1e4` だけで、`83133580` は履歴を読んだだけのセッションです。

## 思考テキストを抜き出す

`type=assistant` のメッセージの `content` から、`type=text` のブロックを出現順にすべて拾い、それぞれ先頭に `💬 ` を付けます。

```bash
python3 - <<'PY'
import json
f="2f0ca1e4-1844-4e02-9eaa-6f57e54ac4f5.jsonl"
with open(f) as fp:
    for d in (json.loads(l) for l in fp if l.strip()):
        if d.get("type")=="assistant":
            for b in d["message"]["content"]:
                if isinstance(b,dict) and b.get("type")=="text":
                    print("💬 "+b["text"])
PY
```

末尾の完了まとめ（最後の `text` ブロック）も思考テキストの一部として一緒に拾います。たとえば、ナレーション6個に完了まとめ1個を加えて、💬 が7個になります。

## 解釈メモを抜き出す

解釈メモは `/home/node/.claude_tmp/...md` に書き出されるので、その書き込みツール呼び出しの中身として記録されます。`cat > ....md <<'EOF' ... EOF` という `Bash` の `tool_use`（場合によっては `Write` の `content`）を探し、`# 担当コメント` で始まる本文を取り出します。

```bash
python3 - <<'PY'
import json
f="2f0ca1e4-1844-4e02-9eaa-6f57e54ac4f5.jsonl"
with open(f) as fp:
    for d in (json.loads(l) for l in fp if l.strip()):
        if d.get("type")=="assistant":
            for b in d["message"]["content"]:
                if isinstance(b,dict) and b.get("type")=="tool_use":
                    s=b.get("input",{}).get("command","")+b.get("input",{}).get("content","")
                    if "# 担当コメント" in s:
                        print(s)
PY
```

見出しの文言は `# 担当コメント XXX の解釈` だったり `# 担当コメント解釈メモ` だったりと揺れます。`の解釈` を条件に入れると後者をすり抜けてしまうので、`# 担当コメント` のプレフィックスだけで引っ掛けます。

なお、解釈メモを書かないセッションもあります（短い応答で、思考の中だけで済ませた場合など）。その場合は、思考テキストだけをまとめ、解釈メモ部分は無しで構いません。

## details に組み立てる

[見本 4720255334](https://github.com/MirrgieRiana/xarpite/issues/550#issuecomment-4720255334) と同じ形にします。本文の末尾、`---` のあと（前後に空行を置きます）に、次のように置きます。

```
---

<details>
<summary>🌱Claude Fairy (Model: … / Protocol 103F vXXX)</summary>

💬 （思考テキスト1）
💬 （思考テキスト2）
…

# 担当コメント解釈メモ …
（解釈メモ全文）

</details>
```

`<summary>` には元のフッター行をそのまま入れます。開くと、思考テキスト → 空行 → 解釈メモ全文、の順に出てくる形です。

## まとめ（最短手順）

1. `grep -l 'あなたの担当コメントは ID <ID> です' *.jsonl` で候補を出します。
2. 宣言行が冒頭システムプロンプトにあるセッションが本物です（`tool_result` だけのセッションは除外します）。
3. `assistant` の `text` ブロックをすべて 💬 付きで拾い、思考テキストにします。
4. `.claude_tmp/...md` を書く `tool_use` の中の `# 担当コメント…` を、解釈メモにします。
5. `<details>` に畳み、対象コメント自身のフッターを差し替えます。
