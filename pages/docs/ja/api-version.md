# APIバージョン

Xarpiteには、Xarpite自身のバージョンとは別の軸として、APIバージョンという概念があります。

APIバージョンは、破壊的変更の世代を表す整数です。

## APIバージョンとは

APIバージョンは、スクリプトが「どの世代までの破壊的変更を取り込んだ環境を想定して書かれているか」を表す整数です。

APIバージョンは文法や組み込みマウント等の動作に影響を与え、ランタイム全体に対して適用されます。

スクリプトが想定するAPIバージョンとランタイムのAPIバージョンが完全に一致しない限り、動作は保証されません。

## 破壊的変更の世代

Xarpiteには、後方互換性を壊す変更が時折加わります。

互換性を壊す変更をAPIバージョンという整数で世代として管理することで、古いスクリプトを古い世代の挙動のまま動かしたり、新しい挙動へ任意のタイミングで移行したりできます。

破壊的変更には、たとえば次のようなものが該当します。

- 関数の削除
- 既存の挙動の変更
- 文法の変更

## スクリプトでのAPIバージョンの扱い

ランタイムのAPIバージョンは `-A` オプションで指定します。

`-A` を指定しない場合、APIバージョンはXarpite自身のメジャーバージョンと同じ値になります。

`-A` の詳細はコマンドラインツールのページを参照してください。

---

現在のランタイムのAPIバージョンは、 `API_VERSION` 定数で取得できます。

```shell
$ xa -A 4 'API_VERSION'
# 4
```

---

移植性を重視するスクリプトは、先頭で `API()` 関数を使って想定するAPIバージョンを表明しておくと安全です。

```shell
$ xa -A 4 -q '
  API(4)
  OUT << "Hello"
'
# Hello
```

想定と異なる世代の環境で実行されたとき、 `API()` は黙って挙動が変わる代わりに、実行のごく早い段階で明示的にエラーをスローします。

```shell
$ xa -A 4 -q '
  API(5)
  OUT << "Hello"
'
# ERROR: Script requires API version 5, but the environment API version is 4
#   at -:2:6             API(5)
#   at -:1:1
```

## なぜ厳密一致なのか

`API()` のチェックは、「指定した値以上」ではなく「指定した値と厳密に一致」で行われます。

破壊的変更は前方にも後方にも互換性を壊しうるためです。

たとえばAPIバージョン5で関数F1が削除され、APIバージョン6で関数F2が削除されたとします。

このときAPIバージョン6の環境でAPIバージョン5を想定したスクリプトを動かすと、F2が既に存在しないため正しく動きません。

逆にAPIバージョン5の環境でAPIバージョン6を想定したスクリプトを動かしても、想定した挙動と食い違います。

したがって、「ちょうどこの世代を想定している」という表明を厳密一致で行うことが、唯一安全な方法になります。

## 破壊的変更とマイグレーション

破壊的変更は、APIバージョンによってオプトインできる形で導入されます。

破壊的変更が影響する機能のドキュメントには、APIバージョンによって挙動がどう変わるかが記述されます。

また、世代をまたいでスクリプトを移行するための案内として、マイグレーションガイドが提供されます。

マイグレーションガイドは、既存の記法・呼び出し・イディオムを新しい世代に向けてどのように書き換えればよいかを案内します。

## 現状

現在ランタイムが提供するAPIバージョンは4のみで、デフォルト値も4です。
